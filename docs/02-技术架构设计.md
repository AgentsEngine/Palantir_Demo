## 1. 架构总览

### 1.1 设计哲学

ManuFoundry 的架构设计直接对标 Palantir Foundry 的五层分层架构。Palantir 之所以能在数据集成与分析领域建立极高的壁垒，核心不在于某一层的单点技术，而在于五层之间的**垂直整合**——数据从接入到决策的完整链路被一套统一的语义模型（Ontology）贯穿。ManuFoundry 将这一理念移植到制造业场景，每一层都针对工厂数据特征做了专门适配。

五层架构遵循严格的**单向依赖原则**：上层可以依赖下层，下层绝不感知上层。这保证了任意层的替换或升级不会引发连锁反应。

### 1.3 各层职责与边界

| 层级 | 名称 | 核心职责 | 对外暴露方式 | 关键设计约束 |
|------|------|----------|-------------|-------------|
| L5 | Workflow | 将分析结果转化为可操作的人机决策流程 | React 前端 + WebSocket 推送 | 仅依赖 L4 的分析结果，不直接查库 |
| L4 | AI/Analytics | 基于本体模型进行预测、诊断、优化 | REST API + 事件订阅 | 模型输入必须经过 L3 本体映射 |
| L3 | Ontology | 统一定义制造业实体、关系、动作的语义模型 | SDK + GraphQL | 唯一的数据语义真相源 (Single Source of Truth) |
| L2 | Data Integration | 从异构系统抽取数据，清洗后写入本体结构 | 连接器 SPI + 管线 DSL | 数据落地必须经过 L3 映射，禁止直写 |
| L1 | Infrastructure | 提供计算、存储、网络的容器化运行环境 | 服务端口 + 卷挂载 | 无状态服务，数据全部持久化到卷 |

### 1.4 为什么 Ontology 是核心护城河

Palantir 的 Ontology 层是其在数据平台领域最难以复制的能力。它不是简单的数据字典或 ORM 映射，而是将业务语义直接嵌入数据层的**活文档**：

- **实体定义**：将"设备""工单""批次"等制造业概念建模为一等公民，而非数据库表名
- **关系建模**：设备之间的上下游依赖、工单与批次的关联关系在图谱中显式表达
- **虚拟属性**：在原始数据上叠加计算逻辑（如"设备健康度 = f(温度, 振动, 历史故障)"），分析层无需理解底层计算
- **动作函数**：Ontology 上可以注册操作（如"下发维修工单""调整产线参数"），将数据分析直接闭环到执行
- **权限绑定**：权限控制粒度精确到实体类型+关系类型，而非粗粒度的表级权限

这使得 ManuFoundry 的每一层都建立在统一的语义基础之上，上层无需关心数据来自 MES 还是 ERP，只要 Ontology 层正确映射，一切实体都以统一的制造业语言存在。

### 3.3 实时数据流 vs 批量数据流

| 维度 | 实时数据流 (IoT/SCADA) | 批量数据流 (ERP/MES/QMS) |
|------|------------------------|--------------------------|
| 触发方式 | 事件驱动 (MQTT/WebSocket) | 定时调度 (Cron) + 手动触发 |
| 数据频率 | 毫秒级 ~ 秒级 | 分钟级 ~ 天级 |
| 缓冲机制 | Redis Stream + 批量写入 | 文件暂存 + 批量导入 |
| 清洗策略 | 流式规则 (滑动窗口) | SQL Transform + 规则引擎 |
| 存储目标 | TimescaleDB 连续聚合 | PostgreSQL 批量 UPSERT |
| 延迟要求 | < 5 秒 (报警) / < 30 秒 (看板) | < 15 分钟 |

### 4.2 PostgreSQL 数据库设计

#### 4.2.1 核心业务表

```sql
-- 用户与权限
CREATE TABLE users (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    username    VARCHAR(64) UNIQUE NOT NULL,
    email       VARCHAR(255) UNIQUE NOT NULL,
    hashed_pwd  VARCHAR(255) NOT NULL,
    role        VARCHAR(32) NOT NULL DEFAULT 'operator',  -- admin|engineer|operator|viewer
    factory_id  UUID REFERENCES factories(id),
    is_active   BOOLEAN DEFAULT TRUE,
    created_at  TIMESTAMPTZ DEFAULT NOW(),
    updated_at  TIMESTAMPTZ DEFAULT NOW()
);

-- 工厂与产线
CREATE TABLE factories (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name        VARCHAR(128) NOT NULL,
    code        VARCHAR(32) UNIQUE NOT NULL,   -- 工厂编码
    location    JSONB,                         -- {province, city, lat, lng}
    timezone    VARCHAR(64) DEFAULT 'Asia/Shanghai',
    metadata    JSONB DEFAULT '{}'
);

CREATE TABLE production_lines (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    factory_id  UUID NOT NULL REFERENCES factories(id),
    name        VARCHAR(128) NOT NULL,
    code        VARCHAR(32) NOT NULL,
    line_type   VARCHAR(32),                   -- assembly|machining|painting|...
    status      VARCHAR(16) DEFAULT 'active',  -- active|maintenance|stopped
    metadata    JSONB DEFAULT '{}'
);

-- 设备主数据
CREATE TABLE equipments (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    line_id     UUID NOT NULL REFERENCES production_lines(id),
    name        VARCHAR(128) NOT NULL,
    code        VARCHAR(32) NOT NULL,          -- 设备编号 CNC-042
    model       VARCHAR(64),
    manufacturer VARCHAR(128),
    install_date DATE,
    specs       JSONB DEFAULT '{}',            -- 设备规格参数
    ontology_id VARCHAR(128),                  -- Neo4j 中对应节点 ID
    status      VARCHAR(16) DEFAULT 'running',
    UNIQUE(line_id, code)
);
```

#### 4.2.2 TimescaleDB 时序表

```sql
-- 启用 TimescaleDB 扩展
CREATE EXTENSION IF NOT EXISTS timescaledb;

-- 传感器读数 (高频写入、批量查询)
CREATE TABLE sensor_readings (
    time          TIMESTAMPTZ NOT NULL,
    equipment_id  UUID NOT NULL,
    sensor_code   VARCHAR(64) NOT NULL,        -- temperature|vibration|pressure|...
    value         DOUBLE PRECISION NOT NULL,
    quality       SMALLINT DEFAULT 100,        -- 数据质量 0-100
    metadata      JSONB DEFAULT '{}'
);

-- 转换为超表 (按时间自动分区)
SELECT create_hypertable('sensor_readings', 'time',
    chunk_time_interval => INTERVAL '1 day',
    partitioning_column => 'equipment_id',
    number_partitions => 4
);

-- 连续聚合 (设备每分钟统计)
CREATE MATERIALIZED VIEW equipment_metrics_minute
WITH (timescaledb.continuous) AS
SELECT
    time_bucket('1 minute', time) AS bucket,
    equipment_id,
    sensor_code,
    AVG(value)   AS avg_value,
    MIN(value)   AS min_value,
    MAX(value)   AS max_value,
    STDDEV(value) AS std_value,
    COUNT(*)     AS sample_count
FROM sensor_readings
GROUP BY bucket, equipment_id, sensor_code;

-- 自动压缩策略 (7 天后压缩，90 天后删除原始数据)
ALTER TABLE sensor_readings SET (
    timescaledb.compress,
    timescaledb.compress_segmentby = 'equipment_id, sensor_code'
);

SELECT add_compression_policy('sensor_readings', INTERVAL '7 days');
SELECT add_retention_policy('sensor_readings', INTERVAL '90 days');
```

## 7. 认证与权限架构

### 7.1 JWT 认证

```
依赖: python-jose (JWT) + passlib (密码哈希)

登录流程:
  1. POST /auth/login {username, password}
  2. 验证密码 → passlib.bcrypt.verify()
  3. 签发 JWT → jose.jwt.encode({sub, uid, exp}, SECRET_KEY)
  4. 返回 {token, user: {id, username, display_name, is_admin, roles}}

前端存储:
  - mf_token → JWT access token
  - mf_user  → JSON.stringify(user)
  - restore() → 页面刷新时从 localStorage 恢复状态
```

### 7.2 RBAC 权限模型

```
User ──N:N──→ Role ──1:N──→ Permission

Permission:
  - resource_type: menu | page | action | data
  - resource_key:  "/" | "/maintenance" | "approve"
  - action:        view | create | edit | delete

前端权限控制:
  1. ROLE_MENU_MAP: 按角色过滤左侧业务菜单
  2. getAdminMenus(): 仅 admin 可见管理页面
  3. isAuthenticated: 路由守卫
  4. 通知 user_id 过滤: 每人只看自己的通知
```

### 6.5 API Key 管理

API Key 用于系统间集成（如 MES 系统调用 ManuFoundry API），与用户 JWT 分离：

```
API Key 格式:  mf_{env}_{random_32chars}
示例:          mf_prod_k8xP2mN9vR3qW7tY5jH1cF4bA6dE0gI

属性:
  - 每个应用/系统一个 Key
  - 可绑定 IP 白名单
  - 可设置权限范围 (scope)
  - 可设置过期时间
  - 支持吊销 (revoked)
  - 使用次数统计与限流

存储:
  - 数据库仅存储 SHA-256 哈希值
  - 明文 Key 仅在创建时返回一次
```

### 6.6 数据安全策略

| 策略 | 实现方式 | 适用范围 |
|------|----------|----------|
| 传输加密 | HTTPS (TLS 1.3) 强制 | 所有 API 请求 |
| 密码存储 | bcrypt (cost=12) | 用户密码 |
| Token 存储 | HttpOnly + Secure Cookie | 前端 Token |
| 敏感字段 | AES-256-GCM 加密 | 数据库中手机号、邮箱等 |
| SQL 注入 | SQLAlchemy 参数化查询 | 所有数据库操作 |
| XSS | CSP Header + 输出编码 | 前端渲染 |
| CSRF | SameSite Cookie + Token | 表单提交 |
| 日志脱敏 | 自动遮盖 PII 字段 | 审计日志 |

### 8.2 规则引擎

```
┌─────────────────────────────────────────────────────────────┐
│                      规则引擎 (Rule Engine)                   │
│                                                             │
│  数据清洗规则:                                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ IF sensor_type == "temperature"                     │   │
│  │    AND value > 200                                  │   │
│  │ THEN mark_as_outlier(reason="超出合理量程")            │   │
│  │      AND notify(equipment_id, severity="warning")   │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  本体映射规则:                                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ WHEN pipeline "mes_import" receives record           │   │
│  │   MAP record["设备编号"] → Equipment.code             │   │
│  │   MAP record["温度"]     → SensorReading.value       │   │
│  │   LINK Equipment -[:HAS_READING]-> SensorReading     │   │
│  │   COMPUTE Equipment.health_score = f(readings)       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  触发规则:                                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ WHEN Equipment.health_score < 0.4                   │   │
│  │   CREATE MaintenanceTask(priority="urgent")          │   │
│  │   NOTIFY Person WHERE role="engineer"                │   │
│  │   LINK MaintenanceTask -[:TARGETS]-> Equipment       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  执行方式:                                                   │
│    - DSL: YAML/JSON 声明式规则定义                            │
│    - 引擎: 自研规则解释器 (支持 AND/OR/NOT 逻辑组合)           │
│    - 触发: 事件驱动 (数据写入时触发) + 定时巡检                 │
│    - 版本: 规则支持版本管理，变更需审批                         │
└─────────────────────────────────────────────────────────────┘
```

### 8.4 系统扩展矩阵

| 扩展维度 | 扩展方式 | 配置化程度 | 是否需要重启 |
|----------|----------|-----------|-------------|
| 新数据源接入 | 注册连接器 | YAML 配置 | 不需要 |
| 数据清洗规则 | 规则引擎 DSL | YAML/JSON | 不需要 |
| 本体实体类型 | Neo4j Schema 扩展 | API 调用 | 不需要 |
| 分析模型 | Model Registry | MLflow CLI | 不需要 (热加载) |
| 仪表盘组件 | React 组件库 | JSON Schema | 刷新前端 |
| 通知渠道 | Notification Plugin | YAML 配置 | 不需要 |
| 权限规则 | RBAC 策略 | API 调用 | 不需要 |
| 存储容量 | Volume 扩容 / 云存储 | 基础设施配置 | 需要重启 |

### 8.5 性能设计目标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| API 响应延迟 (P50) | < 100ms | 简单查询 |
| API 响应延迟 (P99) | < 500ms | 复杂聚合查询 |
| 传感器数据写入吞吐 | > 10,000 点/秒 | 批量写入 TimescaleDB |
| 实时看板刷新延迟 | < 3 秒 | WebSocket 推送 |
| 管线数据处理延迟 | < 15 分钟 | 批量管线端到端 |
| 图谱查询响应 | < 2 秒 | 三度关系遍历 |
| 模型推理延迟 | < 200ms | ONNX Runtime |
| 系统可用性 | > 99.5% | 单机部署目标 |

## 附录 B: 关键架构决策记录 (ADR)

| 编号 | 决策 | 背景 | 结论 | 理由 |
|------|------|------|------|------|
| ADR-001 | 时序数据库选型 | IoT 数据需要高性能时序存储 | 选择 TimescaleDB 而非 InfluxDB | 与 PostgreSQL 同生态，减少运维复杂度，避免引入新数据库类型 |
| ADR-002 | 图数据库选型 | Ontology 层需要表达复杂实体关系 | 选择 Neo4j 而非 NebulaGraph | Cypher 语言表达力强，社区成熟，制造业知识图谱有大量参考案例 |
| ADR-003 | 后端语言选型 | 需要 AI/ML 生态 + 高性能 API | 选择 Python (FastAPI) 而非 Go/Java | AI/ML 生态无可替代，FastAPI async 性能已满足 10K QPS 目标 |
| ADR-004 | 前端框架选型 | 需要复杂图谱可视化 + 企业级组件 | 选择 React 而非 Vue | ECharts/Cytoscape/ReactFlow 的 React 绑定更成熟，Ant Design Pro 开箱即用 |
| ADR-005 | 初始部署方式 | 团队规模小，需快速验证 | Docker Compose 单机部署 | 满足初期验证需求，所有服务保持 K8s 兼容设计，迁移成本极低 |
| ADR-006 | 认证方案 | 需要无状态认证以支持水平扩展 | JWT + Redis 会话 | 无状态特性天然支持多实例部署，Redis 弥补 JWT 的吊销短板 |
