## 1. 系统分层

ManuFoundry 采用经典 N 层 + 语义中间层的混合架构。从下到上：

| 层 | 职责 | 主要技术 |
|---|---|---|
| **存储层** | 关系数据 / 图数据 / 缓存 / 大对象 | PostgreSQL、Neo4j、Redis、文件系统 |
| **语义层** | 本体（Ontology）+ 元模型，统一业务实体语义 | `app/models/graph_models.py`、`app/models/relational.py`、`MetaModel/MetaField` |
| **数据集成层** | 数据源接入、清洗、转换、写入 PG/Neo4j | `app/services/data_integration/` |
| **分析层** | 预测性维护、SPC、供应链优化等领域算法 | `app/services/analytics/` |
| **API 层** | FastAPI 路由 + Pydantic schema + Depends 鉴权 | `app/api/*` |
| **前端层** | React + Antd + Zustand + Vite + ECharts/Cytoscape | `frontend/src/*` |

层与层之间**只允许向下依赖**：API 层可以调 service 层、模型层；service 层只调模型层与外部驱动；模型层不引用任何 API/service 代码。

## 3. 模块依赖拓扑

backend 内部的依赖图（仅列关键边，避免噪声）：

```
                            app.config.Settings
                                   ▲
                                   │ (env)
                                   │
              ┌────────────────────┼─────────────────────────┐
              │                    │                         │
       app.core.logging      app.core.security          app.database
              ▲                    ▲                         ▲
              │                    │                         │
              └────── app.core.db ─┘                         │
                          ▲                                  │
                          │ db_session / safe_db_call        │
                          │                                  │
                  app.api.deps  ─────────────────────────────┘
                          ▲
                          │ Depends(get_current_user) / Depends(get_db)
                          │
            ┌─────────────┼─────────────────────────────────┐
            │             │                                 │
       app.api.auth   app.api.<biz_router>          app.api.model_driven
                          │                                 │
                          │                       ┌─────────┴────────┐
                          │                       │                  │
                          │             _model_driven_shared    model_driven_meta
                          ▼                       │                  │
                     app.models.*                 │             model_driven_data
                                                  │                  │
                                                  └──── safe_db_call ┘
```

要点：

- `app.core.*` 是叶子模块，**不允许反向依赖** `app.api` / `app.services` / `app.models`。
- `app.api.deps.get_db` 透过 `app.core.db.db_session` 提供 `AsyncSession`，是受保护路由的唯一 DB 注入入口。
- 所有 router 通过 `from app.core.db import safe_db_call as _try_db` 共用同一个错误处理管道（log + None fallback）。
- `model_driven` 由四个文件组成：`_model_driven_shared` 提供共享原语，`model_driven_meta/data/menus` 各自挂载子 router，`model_driven.py` 通过 `APIRouter.include_router()` 聚合，对外暴露单一 `router` 符号供 `main.py` 引用。

## 4. 运行时数据流

以「前端加载 `/dashboard/overview`」为例：

```
┌─ Browser
│   1. page mount → api.getOverview()
│   2. axios request interceptor 注入 Authorization: Bearer <token>
└──┬─────────────────────────────────────────────────────────────
   │ HTTPS / Vite proxy / nginx
   ▼
┌─ FastAPI (main.py)
│   3. CORSMiddleware  → Authorization 通过
│   4. Route 匹配 /api/v1/dashboard/overview
│   5. Depends(get_current_user) 解码 JWT
│   6. Depends(get_db) 通过 db_session() 拿 AsyncSession
└──┬─────────────────────────────────────────────────────────────
   │
   ▼
┌─ Router 业务函数 (dashboard.py)
│   7. await _try_db(_query)
│        ├─ try: 真实 PG 查询
│        └─ except: log warning + return None → fallback to mock
│   8. 返回 JSON
└──┬─────────────────────────────────────────────────────────────
   │
   ▼
┌─ Browser
│   9. axios response 解析 res.data
│  10. 渲染 ECharts / Antd Statistic / Table
└────────────────────────────────────────────────────────────────
```

如途中抛未捕获异常：`main.unhandled_exception_handler` 兜底，统一返回 `500 + {detail, type}`，避免 stacktrace 外泄。

## 5. 部署形态

| 形态 | 命令 | 说明 |
|---|---|---|
| **Local dev** | `cd backend && uvicorn app.main:app --reload` + `cd frontend && npm run dev` | 直接连 SQLite fallback，最快 onboarding 路径 |
| **Docker dev** | `docker compose -f docker/docker-compose.yml up -d` | PG/Neo4j/Redis 完整栈 + 后端 reload + 前端 Vite HMR |
| **Docker prod** | `docker compose -f docker/docker-compose.yml -f docker/docker-compose.prod.yml up -d --build` | 后端去掉 `--reload`，前端用 nginx 提供静态 dist；`DEMO_AUTH_OPTIONAL=false` 强制 401 |
| **K8s** | TBD | 当前未提供 helm chart |

详见 `15-部署与运行.md`。

## 6. 关键设计取舍

### 6.1 为什么 SQLite + PostgreSQL 双引擎？

`app/database.py` 在 import 时优先创建 PG 引擎；若 `asyncpg` 缺失则回退到 SQLite + aiosqlite，让 demo 可在零依赖环境跑通。代价是 `init_db()` 仅 SQLite 模式建表，PG 必须走 Alembic — 这点已在 `database.init_db` 注释明示。

### 6.2 为什么所有 router 都有 `_try_db + mock fallback`？

这是 demo 项目的**显式取舍**：让前端在 PG/Neo4j 都未就绪的情况下仍能展示完整 UI。代价是错误被吞为 `warning` 级别 log。生产部署应：

- 设 `LOG_LEVEL=DEBUG` 短期观察 fallback 频次；
- 对关键业务接口（如 auth、admin）改为直接抛 5xx 而不是返回 mock。

### 6.3 为什么 model_driven 拆四个文件而非保留 643 行单文件？

- 三类职责（meta / data / menus）的修改频率与作者域不同；
- 动态 SQL CRUD 是高安全风险点，独立成 `model_driven_data.py` 便于代码审计；
- mock 与 schema 复用通过 `_model_driven_shared` 收敛，避免重复声明。

### 6.4 为什么前端不用 Redux Toolkit？

- 仅需鉴权一处全局状态，Zustand 体积更小（~1KB）；
- 业务页面间没有共享数据流，Antd `useState` + `useMemo` 已足够。
