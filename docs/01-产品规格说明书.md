## 1. 产品愿景与定位

### 1.1 一句话定义

**ManuFoundry —— 制造业的数据操作系统。**

### 1.2 愿景陈述

正如 Palantir Foundry 为政府、金融、医疗等行业提供了统一的数据集成与分析操作系统，ManuFoundry 致力于为离散制造业和流程制造业构建一个**全量数据汇聚、全链路可追溯、全场景可决策**的数据操作系统。

制造业的核心矛盾是：产线上每天都在产生海量数据（PLC 传感器、MES 工单、ERP 订单、SCADA 报警、质量检测图像），但这些数据分散在不同系统中，形成一座座"数据烟囱"。ManuFoundry 的使命是**打通这些数据壁垒，让数据在制造全价值链中流动，驱动从"经验决策"到"数据决策"的根本性转变**。

### 1.4 与 Palantir Foundry 的对标关系

| 维度 | Palantir Foundry | ManuFoundry |
|------|-------------------|-------------|
| **目标行业** | 政府、金融、国防、医疗 | 离散制造、流程制造 |
| **核心数据源** | 情报数据、交易记录、开源情报 | PLC/SCADA 传感器、MES 工单、ERP 订单、QMS 质检 |
| **数据集成** | 多源异构数据融合 | 工业协议（OPC-UA/Modbus/MQTT）+ IT 系统集成 |
| **分析能力** | 全源情报分析、反欺诈 | OEE 分析、预测性维护、SPC 统计过程控制 |
| **协作模式** | 跨机构数据共享 | 跨工厂/跨产线数据共享与对标 |
| **部署方式** | 私有化/Apollo SaaS | 边缘-云混合架构 |
| **商业模式** | 年度平台许可 + 专业服务 | SaaS 订阅 + 实施服务 + 边缘节点许可 |

### 1.5 产品原则

1. **数据不动，计算动**：支持在数据所在的位置执行计算（边缘端/工厂内），仅将聚合结果上云
2. **语义层统一**：所有数据源映射到统一的制造语义模型（设备模型、工艺模型、物料模型）
3. **场景驱动，而非功能堆叠**：每个功能模块必须对应一个可衡量的业务场景
4. **渐进式部署**：从单场景切入，逐步扩展，不要求一次性全厂上线
5. **开放可扩展**：提供标准 API 和插件机制，允许第三方和客户自定义扩展

### 2.2 核心用户详细画像

#### 画像 A：工厂数字化负责人（Digital Transformation Lead）

### 4.2 逐层详细映射

#### Layer 1：数据源连接层

| 维度 | Palantir Foundry | ManuFoundry |
|------|-------------------|-------------|
| **定位** | 连接企业内外部数据源 | 连接工厂 OT 系统和 IT 系统 |
| **协议支持** | JDBC、REST API、文件系统、S3 | OPC-UA、Modbus TCP/RTU、MQTT、S7、HTTP REST、JDBC |
| **数据类型** | 结构化数据、文档、图像 | 时序数据（传感器）、结构化数据（MES/ERP）、图像（质检）、日志 |
| **连接器** | SAP、Salesforce、Snowflake 等预置连接器 | 西门子 S7、三菱 MC、欧姆龙 FINS、SAP、用友、金蝶等预置连接器 |
| **边缘能力** | 无原生边缘能力 | 边缘网关支持协议转换、数据过滤、本地缓存 |
| **ManuFoundry 关键能力** | — | 支持离线采集（断网缓存 + 断点续传）；支持数据采集模板，5 分钟配置一个新的数据源 |

#### Layer 2：数据管道层

| 维度 | Palantir Foundry | ManuFoundry |
|------|-------------------|-------------|
| **定位** | 数据清洗、转换、融合的管道 | 工业数据的清洗、对齐、融合管道 |
| **核心挑战** | 多源数据 schema 对齐 | 多采样率对齐、时间戳同步、设备编码统一 |

| **数据质量** | 数据质量规则引擎 | 数据质量规则 + 传感器数据专用校验（量程检查、跳变检测、死值检测） |

## 6. 低代码平台能力

### 6.1 报表中心（Phase 1）

拖拽式报表编辑器，用户从组件面板拖出图表/KPI卡/数据表到画布，配置数据源后保存，下次打开自动恢复。

**报表配置 JSON Schema：**

```json
{
  "canvas": { "gridSize": 8 },
  "widgets": [
    {
      "id": "w-1", "type": "kpi-card", "title": "设备总数",
      "position": { "x": 0, "y": 0, "w": 6, "h": 4 },
      "dataSource": { "endpoint": "/api/v1/dashboard/overview", "path": "equipment.total" },
      "style": { "color": "#1677ff", "icon": "ToolOutlined" }
    }
  ],
  "filters": [{ "id": "f-1", "type": "date-range", "label": "日期范围" }]
}
```

**支持的组件类型（7 种）：**

| 组件 | 说明 | 数据绑定 |
|------|------|---------|
| KPI 指标卡 | 单值+趋势箭头 | endpoint + path |
| 折线图 | 多序列时序 | endpoint + chartConfig.series |
| 柱状图 | 分类对比 | endpoint + chartConfig.series |
| 饼图 | 占比分布 | endpoint + chartConfig |
| 仪表盘 | 区间指标 | endpoint + path + ranges |
| 数据表格 | 分页列表 | endpoint + columns |
| 文本标注 | 静态内容 | 无 |

**技术方案：** CSS Grid + HTML5 Drag API（无第三方依赖），ReactECharts 图表渲染，Widget Registry 模式支持扩展。

### 6.2 模型驱动页面（Phase 2）

系统配置中心，定义元模型（实体+字段+关系），选择范式自动生成 CRUD 页面，动态注册到导航菜单。

**核心数据模型：**

```
MetaModel（元模型）
  ├── MetaField（字段定义）
  │     ├── field_type: text/number/date/select/relation
  │     ├── visible_in_list / visible_in_form / searchable / sortable
  │     └── relation_config / enum_values
  └── MetaRelation（关系定义）
        ├── relation_type: one_to_one / one_to_many / many_to_many
        └── foreign_key_field

PageConfig（页面配置）
  ├── paradigm: master_detail / form_flow / kanban / tree / graph / report
  ├── model_id → MetaModel
  ├── config (JSON) → 列配置/筛选器/表单布局
  └── route_path → /dynamic/{slug}

MenuItem（动态菜单）
  ├── parent_id → 支持树形层级
  └── is_visible / sort_order
```

**6 种页面范式：**

| 范式 | 说明 | 适用场景 |
|------|------|---------|
| 主从列表 (master_detail) | 左列表右详情 | 设备管理、供应商管理 |
| 表单流程 (form_flow) | 表单+审批流程 | 维修申请、质量处理 |
| 看板仪表盘 (kanban) | 拖拽式卡片看板 | 工单管理、任务分配 |
| 树形层级 (tree) | 可展开树结构 | 组织架构、BOM管理 |
| 关系图谱 (graph) | 节点+边的网络图 | 供应链关系、故障链路 |
| 报表引擎 (report) | 嵌入报表中心 | 各类分析报表 |

**关键能力：** 本体一键导入（ENTITY_SCHEMAS → MetaModel），动态数据 CRUD（白名单防注入），通配路由 `/dynamic/:slug`。

│  └──────────────────────────────────────────────────┘       │
│      │                                                      │
│      ▼                                                      │

### 7.2 RBAC 权限模型

```
User ──→ UserRole ──→ Role ──→ RolePermission
                                   │
                                   ├── resource_type: menu / page / action / data
                                   ├── resource_key: "/" / "/maintenance" / "approve"
                                   └── action: view / create / edit / delete
```

| 层面 | 实现方式 | 说明 |
|------|---------|------|
