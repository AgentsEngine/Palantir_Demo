## 1. 本体设计理念

### 1.1 为什么 Ontology 是护城河

Palantir Foundry 的 Ontology 不是一张简单的 ER 图或数据字典。它是一个**活的语义层**——所有业务实体、关系、动作在 Ontology 中有且仅有一个权威定义，上层应用（看板、工作流、AI 分析）全部基于这个语义层构建，而不是直接查询原始表。

这意味着：**换掉任何一个上层应用，底层语义不受影响；换掉任何一个数据源，上层应用看到的世界不变。** 这就是 Palantir 最深的壁垒。ManuFoundry 将这一理念完整移植到制造业场景。

### 1.2 四大设计原则

| 原则 | Palantir 原意 | ManuFoundry 制造业诠释 |
|------|--------------|----------------------|
| **业务实体中心** | 一切围绕业务对象（Object）建模，而非围绕表或文件 | 生产线上的设备、工单、产品不是数据库表的一行，而是有身份、有属性、有行为的实体 |
| **关系一等公民** | 关系（Link）与实体同等重要，可独立查询、授权、加动作 | "设备A安装在产线B上"不是 JOIN 条件，而是一条可追溯、可授权、带时间戳的关系 |
| **时间旅行** | 每个实体和关系保留完整变更历史，可回溯到任意时间点 | "这台设备上周二的状态是什么""这条工艺路线改过几次"——制造合规审计的刚需 |
| **权限粒度** | 权限绑定到实体实例和属性级别，而非表级别 | 车间主任只能看本车间的设备数据，质量工程师能看到缺陷但不能改工艺参数 |

### 1.3 Ontology 在五层架构中的位置

```
Layer 5: Workflow    ── 所有操作界面调用 Ontology Action
Layer 4: AI/Analytics ── 所有分析查询基于 Ontology View
Layer 3: Ontology    ══ ★ 你正在看这一层 ★
Layer 2: Integration ── Pipeline 输出映射到 Ontology Entity
Layer 1: Infra       ── 不感知 Ontology
```

Ontology 是唯一一个同时被上层四层依赖的核心层。它的设计质量直接决定了整个系统的上限。

## 2. 实体类型定义（6 大族）

制造业本体包含 **6 大实体族（Family）**、**22 种实体类型**。每个族对应一个独立的业务领域，族与族之间通过关系连接。

### 2.1 工厂族 — Factory Family

工厂族描述物理世界的生产设施层级：工厂 → 车间 → 产线 → 设备 → 传感器。严格遵循 ISA-95 设备层级模型。

## 3. 关系类型定义（7 种关系）

关系（Relationship / Link）在本体中是一等公民。每条关系都有唯一 ID、时间戳、属性和权限控制。

### 3.1 关系总览

| 关系类型 | 源实体 | 目标实体 | 语义 | 典型场景 |
|----------|--------|----------|------|----------|
| **CONTAINS** | Factory → Workshop → ProductionLine → Equipment → Sensor | 层级子节点 | 物理包含/层级结构 | 工厂包含车间，车间包含产线 |
| **PRODUCES** | WorkOrder → Product, ProductionLine → Product | 产出物 | 生产了什么 | 工单生产某型号产品 |
| **REQUIRES** | BOM → Material, WorkOrder → Equipment, Operation → Worker | 依赖物 | 生产所需的资源 | BOM 需要某种原材料 |
| **SUPPLIES** | Supplier → Material, Shipment → Warehouse | 供应关系 | 谁提供什么 | 供应商提供原材料 |
| **INSPECTS** | Inspection → Defect, Worker → SPCPoint | 检验关系 | 检查了什么 | 检验单发现缺陷 |
| **MAINTAINS** | Worker → Equipment, CAPA → Defect | 维护/纠正 | 谁维护什么 | 技术员维护设备 |
| **FEEDS** | Sensor → Equipment, Sensor → SPCPoint | 数据流 | 传感器数据流入 | 温度传感器为 SPC 提供数据 |

### 3.2 关系属性（所有关系共有）

每条关系都附带以下基础属性，存储在关系节点上：

| 属性名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| rel_id | STRING | Y | 关系唯一 ID（UUID） |
| effective_from | DATETIME | Y | 关系生效时间 |
| effective_to | DATETIME | N | 关系失效时间（NULL 表示仍然有效） |
| confidence | FLOAT | N | 置信度（AI 推理出的关系，0~1） |
| source | STRING | N | 关系来源（MANUAL / PIPELINE / AI_INFERRED） |

### 3.3 各关系的专属属性

#### CONTAINS — 特有属性

### 4.4 五个常用 Cypher 查询

#### 查询 1：设备全链路追溯 — "某设备关联的所有工单和缺陷"

```cypher
// 场景：设备故障时，快速定位影响了哪些工单、产生了哪些缺陷
MATCH (e:Equipment {equipment_id: $equipment_id})<-[:CONTAINS]-(line:ProductionLine)
MATCH (wo:WorkOrder)-[:REQUIRES]->(e)
OPTIONAL MATCH (wo)-[:PRODUCES]->(p:Product)
OPTIONAL MATCH (op:Operation {equipment_id: e.equipment_id})<-[:REQUIRES]-(wo2:WorkOrder)
OPTIONAL MATCH (ins:Inspection)-[:INSPECTS]->(d:Defect)
  WHERE ins.wo_id = wo.wo_id OR ins.wo_id = wo2.wo_id
RETURN e.equipment_id AS equipment,
       e.status AS equipment_status,
       e.health_score AS health,
       collect(DISTINCT {
         wo: wo.wo_id,
         product: p.product_id,
         status: wo.status,
         planned_qty: wo.planned_qty,
         completed_qty: wo.completed_qty
       }) AS work_orders,
       collect(DISTINCT {
         defect: d.defect_id,
         type: d.defect_type,
         severity: d.severity,
         status: d.status
       }) AS defects
ORDER BY e.health_score ASC;
```

#### 查询 2：产品族谱展开 — "某产品的完整 BOM + 工艺路线"

```cypher
// 场景：新产品导入或工程变更时，查看完整物料需求和工艺步骤
MATCH (p:Product {product_id: $product_id})
MATCH (bom:BOM {product_id: p.product_id, status: 'APPROVED'})
MATCH (bom)-[:REQUIRES]->(m:Material)
MATCH (route:ProcessRoute {product_id: p.product_id, status: 'APPROVED'})
RETURN p.name AS product_name,
       bom.version AS bom_version,
       collect(DISTINCT {
         material: m.name,
         material_id: m.material_id,
         type: m.material_type,
         current_stock: m.current_stock,
         safety_stock: m.safety_stock
       }) AS materials,
       route.version AS route_version,
       route.steps AS process_steps,
       route.total_cycle_sec AS total_cycle_sec;
```

#### 查询 3：供应商风险传导 — "某供应商断供影响哪些产品和订单"

```cypher
// 场景：供应商出问题时，评估影响范围
MATCH (sup:Supplier {supplier_id: $supplier_id})-[:SUPPLIES]->(m:Material)
MATCH (bom:BOM)-[:REQUIRES]->(m)
MATCH (p:Product)<-[:PRODUCES]-(wo:WorkOrder)
  WHERE bom.product_id = p.product_id
    AND wo.status IN ['CREATED', 'RELEASED', 'IN_PROGRESS']
MATCH (so:SalesOrder)
  WHERE so.order_id = wo.sales_order_id
RETURN sup.name AS supplier,
       m.name AS material,
       m.current_stock AS remaining_stock,
       m.lead_time_days AS lead_time,
       collect(DISTINCT p.name) AS affected_products,
       collect(DISTINCT {
         order: so.order_id,
         customer: so.customer_id,
         delivery: so.delivery_date,
         status: so.fulfillment_status
       }) AS at_risk_orders;
```

#### 查询 4：质量异常根因定位 — "某缺陷类型的高频设备和方法"

```cypher
// 场景：缺陷持续出现时，定位根因设备和工序
MATCH (d:Defect {defect_type: $defect_type, status: 'OPEN'})
MATCH (ins:Inspection)-[:INSPECTS]->(d)
OPTIONAL MATCH (ins)-[:INSPECTS]->(op:Operation)
OPTIONAL MATCH (op)-[:REQUIRES]->(e:Equipment)
OPTIONAL MATCH (op)-[:REQUIRES]->(w:Worker)
RETURN d.defect_type,
       d.severity,
       count(d) AS defect_count,
       collect(DISTINCT e.equipment_id) AS suspect_equipment,
       collect(DISTINCT op.operation_type) AS suspect_operations,
       collect(DISTINCT w.worker_id) AS suspect_operators,
       avg(d.quantity) AS avg_defect_qty_per_instance
ORDER BY defect_count DESC
LIMIT 20;
```

#### 查询 5：车间 OEE 实时仪表盘

```cypher
// 场景：管理看板，实时监控车间每条产线的状态和关键指标
MATCH (ws:Workshop {workshop_id: $workshop_id})-[:CONTAINS]->(line:ProductionLine)
MATCH (line)-[:CONTAINS]->(e:Equipment)
OPTIONAL MATCH (line)-[:PRODUCES]->(p:Product)
OPTIONAL MATCH (wo:WorkOrder)-[:REQUIRES]->(e)
  WHERE wo.status = 'IN_PROGRESS'
RETURN line.line_id AS line,
       line.name AS line_name,
       line.status AS status,
       line.cycle_time_sec AS cycle_time,
       line.oee_target AS oee_target,
       count(e) AS equipment_count,
       count(e { .status: 'RUNNING' }) AS running_count,
       count(e { .status: 'FAULT' }) AS fault_count,
       avg(e.health_score) AS avg_health,
       collect(DISTINCT wo.wo_id) AS active_work_orders,
       collect(DISTINCT p.product_id) AS producing_products
ORDER BY line.line_id;
```

## 5. 时间旅行设计

时间旅行是 Palantir Ontology 的标志性能力。在制造业场景中，它直接服务于**合规审计（ISO / IATF）、质量追溯、工艺变更回溯**等刚需场景。

### 5.1 设计目标

- 查询任意实体在**任意历史时间点**的完整状态
- 查询任意关系在**任意历史时间点**是否存在及其属性
- 支持审计日志：谁在什么时候改了什么，从什么值改成了什么值
- 存储成本可控：不保留高频时序数据（那是时序数据库的职责），只保留属性变更

### 5.2 变更日志模型

采用 **Event Sourcing** 模式，所有变更以事件形式追加写入 `ChangeLog` 节点：

```cypher
-- ChangeLog 节点
(:ChangeLog {
  change_id: UUID,              -- 变更唯一 ID
  target_type: STRING,          -- 目标实体类型（如 "Equipment"）
  target_id: STRING,            -- 目标实体 ID
  change_type: ENUM,            -- CREATED / UPDATED / DELETED
  field_name: STRING,           -- 变更的属性名
  old_value: ANY,               -- 变更前值（JSON 序列化）
  new_value: ANY,               -- 变更后值（JSON 序列化）
  changed_by: STRING,           -- 操作人
  changed_at: DATETIME,         -- 变更时间
  reason: STRING,               -- 变更原因
  source: STRING                -- 来源（MANUAL / PIPELINE / ACTION）
})

-- 关系：变更日志指向被变更的实体
(:ChangeLog)-[:APPLIES_TO]->(:Equipment)
(:ChangeLog)-[:APPLIES_TO]->(:WorkOrder)
-- ... 适用于所有实体类型
```

### 5.3 时间点查询语法

ManuFoundry 提供封装好的时间旅行查询函数，底层基于 ChangeLog 节点回溯：

#### 查询实体在某个时间点的状态

```cypher
// 示例：查询设备 EQ-CNC-042 在 2026-03-15 的状态
// 策略：取最新快照，然后回放该时间点之前的所有 ChangeLog

// Step 1: 获取当前最新状态
MATCH (e:Equipment {equipment_id: 'EQ-CNC-042'})
// Step 2: 获取目标时间点之前的所有变更事件，按时间倒序
OPTIONAL MATCH (cl:ChangeLog)-[:APPLIES_TO]->(e)
  WHERE cl.changed_at <= datetime('2026-03-15T23:59:59')
  WITH e, cl
  ORDER BY cl.changed_at DESC
// Step 3: 应用变更还原（应用层逻辑）
RETURN e AS current_state,
       collect(cl) AS changes_up_to_target_date;
```

> 实际实现中，ManuFoundry 在应用层提供 `entityAt(entityId, timestamp)` API，自动完成"取当前值 → 回放 ChangeLog → 返回历史值"的完整逻辑。

#### 查询实体在某个时间段的变更历史

```cypher
// 示例：查询工单 WO-2026-0315 在过去 7 天的所有变更
MATCH (cl:ChangeLog)-[:APPLIES_TO]->(wo:WorkOrder {wo_id: 'WO-2026-0315'})
WHERE cl.changed_at >= datetime() - duration({days: 7})
RETURN cl.changed_at AS time,
       cl.change_type AS type,
       cl.field_name AS field,
       cl.old_value AS old_val,
       cl.new_value AS new_val,
       cl.changed_by AS operator,
       cl.reason AS reason
ORDER BY cl.changed_at ASC;
```

#### 查询关系在某个时间点的状态

```cypher
// 示例：查询 2026-03-01 时设备的 CONTAINS 关系（当时安装在哪个产线上）
MATCH (e:Equipment {equipment_id: 'EQ-CNC-042'})-[r:CONTAINS]-(line:ProductionLine)
WHERE r.effective_from <= datetime('2026-03-01T00:00:00')
  AND (r.effective_to IS NULL OR r.effective_to > datetime('2026-03-01T00:00:00'))
RETURN e.equipment_id, line.line_id, r.position, r.install_date;
```

### 5.4 存储策略

| 数据类型 | 保留策略 | 说明 |
|----------|----------|------|
| 实体属性变更 | 永久保留 | 合规要求，不可删除 |
| 关系变更 | 永久保留 | 供应链追溯要求 |
| ChangeLog 压缩 | > 2 年可压缩 | 同一字段连续相同值合并为一条 |
| 时序传感器数据 | 不在本体存储 | 由 TimescaleDB 管理，本体只存最新聚合值 |

## 6. 权限模型

Palantir 的权限模型粒度极细——不是"谁能访问哪张表"，而是"谁能看到哪个实体的哪个属性"和"谁能在什么条件下建立关系"。ManuFoundry 针对制造业的多租户、多工厂、多角色场景，设计了四级权限控制。

### 6.1 权限层级

```
┌──────────────────────────────────────────────────┐
│  Level 1: 对象级（Object-Level）                   │
│  能否看到这个实体存在？                              │
│  例：车间主任只能看到本车间设备                       │
├──────────────────────────────────────────────────┤
│  Level 2: 属性级（Attribute-Level）                │
│  能否看到这个实体的某个属性？                         │
│  例：操作员能看到设备状态，看不到采购价格              │
├──────────────────────────────────────────────────┤
│  Level 3: 关系级（Relationship-Level）             │
│  能否看到/建立实体之间的关系？                        │
│  例：质量工程师能看到缺陷-设备关系，不能修改 BOM 关系   │
├──────────────────────────────────────────────────┤
│  Level 4: 行级（Row-Level / 条件级）               │
│  在什么条件下可以访问？                              │
│  例：只能看本工厂的数据，只能改自己负责的设备           │
└──────────────────────────────────────────────────┘
```

### 6.2 权限模型定义

#### PermissionRule 节点

```cypher
(:PermissionRule {
  rule_id: UUID,
  name: STRING,                     -- 规则名称
  description: STRING,

  -- 主体（谁能）
  principal_type: ENUM,             -- USER / ROLE / GROUP
  principal_id: STRING,             -- 用户 ID / 角色 ID / 组 ID

  -- 客体（对什么）
  target_entity_type: STRING,       -- 目标实体类型（如 "Equipment"），* 表示全部
  target_entity_filter: STRING,     -- Cypher WHERE 条件表达式

  -- 操作
  action: ENUM,                     -- READ / CREATE / UPDATE / DELETE / LINK / UNLINK

  -- 字段级
  field_restrictions: MAP,          -- {field_name: "MASKED" | "HIDDEN" | "READONLY"}

  -- 条件
  condition: STRING,                -- 动态表达式（如 "entity.factory_id = user.factory_id"）

  -- 优先级
  priority: INT,                    -- 数字越大优先级越高
  effect: ENUM,                     -- ALLOW / DENY

  -- 审计
  created_by: STRING,
  created_at: DATETIME
})
```

### 6.3 权限评估算法

```
输入: (principal, action, target_entity, field?)
输出: ALLOW / DENY

1. 收集所有匹配的 PermissionRule：
   - principal 匹配（USER 精确 / ROLE 包含 / GROUP 包含）
   - target_entity_type 匹配
   - action 匹配

2. 按 priority 降序排列

3. 逐条评估 condition 表达式：
   - 替换 entity.* 为目标实体属性
   - 替换 user.* 为当前用户属性
   - 计算布尔结果

4. 返回第一条匹配规则的 effect（DENY 优先原则）

5. 默认策略：无匹配规则 = DENY
```

### 6.5 权限与 Neo4j 的集成

ManuFoundry 不依赖 Neo4j 原生的角色权限（粒度不够），而是在应用层实现权限拦截：

```
┌────────────────────────────────────────────┐
│            API Gateway                     │
│   1. 认证（JWT）→ 提取 principal           │
│   2. 解析请求 → 提取 action + target       │
│   3. 查询 PermissionRule → 评估权限        │
│   4. 通过 → 注入 Cypher WHERE 条件         │
│      拒绝 → 403                            │
└────────────────┬───────────────────────────┘
                 │ 注入过滤条件
                 ▼
┌────────────────────────────────────────────┐
│            Neo4j Query Layer               │
│   MATCH (e:Equipment)                      │
│   WHERE e.factory_id = $user_factory_id    │  ← 自动注入
│   RETURN e.name, e.status                  │  ← 字段过滤
└────────────────────────────────────────────┘
```

核心思路：**权限评估在 API 层完成，将权限条件转化为 Cypher 的 WHERE 子句注入到查询中**。这样无需维护 Neo4j 的内部用户体系，且权限变更即时生效。
