## 1. 先决条件

| 模式 | 必需 | 推荐版本 |
|---|---|---|
| 本地 (SQLite fallback) | Python 3.11+ / Node 20+ | 3.11.7 / 20.11 |
| 本地 (PG) | 上 + PostgreSQL 16 | 16.2 |
| Docker | Docker Desktop / Docker Engine 24+ + Compose v2 | — |

仓库根有：

- `.env.example` — 复制为 `.env` 后按需修改
- `.gitignore` — 已忽略 `.env`、`*.db`、`node_modules/` 等
- `docker/docker-compose.yml` — 默认 dev 栈
- `docker/docker-compose.prod.yml` — 生产 overlay

## 2. 本地开发模式

### 2.1 后端

```bash
cd backend
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp ../.env.example ../.env       # 可选；不创建则使用 Settings 默认值
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

启动后会出现：

```
[INFO] app.database: Database backend: sqlite
[INFO] app.database: SQLite schema ensured
[INFO] app.database: Seeded N rows
[INFO] uvicorn: Application startup complete.
```

### 2.2 前端

```bash
cd frontend
npm install
npm run dev
```

打开 <http://localhost:3000>，登录页测试账号：

| username | password | 角色 |
|---|---|---|
| admin | admin123 | 超级管理员 |
| zhangsan | 123456 | 生产主管 |
| lisi | 123456 | 质检员 |

### 2.3 切换到 PostgreSQL

启动本机 PG 后在 `.env` 中：

```dotenv
POSTGRES_HOST=localhost
POSTGRES_PORT=5432
POSTGRES_USER=manufoundry
POSTGRES_PASSWORD=...
POSTGRES_DB=manufoundry
```

注意 PG 模式下 `init_db` 不会自动建表，必须先：

```bash
cd backend
alembic upgrade head
python ../scripts/seed_data.py    # 可选：导入种子
```

## 3. Docker 开发模式

```bash
cd <repo-root>
docker compose -f docker/docker-compose.yml up -d --build
```

启动的服务：

| 服务 | 端口 | 容器名 | 备注 |
|---|---|---|---|
| postgres | 5432 | manufoundry-postgres | volume `postgres_data` |
| neo4j | 7474 / 7687 | manufoundry-neo4j | APOC 插件需联网首次下载 |
| redis | 6379 | manufoundry-redis | — |
| backend | 8000 | manufoundry-backend | `Dockerfile.dev` + `uvicorn --reload` + 源码 mount |
| frontend | 3000 | manufoundry-frontend | `Dockerfile.dev` + Vite HMR + 源码 mount |

源码挂载方式：

- `../backend:/app` — 后端整目录
- `../data:/data` — 种子数据（容器内被 `database._seed_sqlite` 读取）
- `../frontend:/app` + `匿名 volume /app/node_modules` — 防止宿主机覆盖容器内 `node_modules`

健康检查：所有服务都有 `healthcheck`，`backend` 在 `postgres / neo4j / redis` 全部 healthy 后才启动（`depends_on.condition: service_healthy`）。

查看日志：

```bash
docker compose -f docker/docker-compose.yml logs -f backend
docker compose -f docker/docker-compose.yml logs -f frontend
```

停机（保留 volume）：

```bash
docker compose -f docker/docker-compose.yml down
```

完全清理（含数据）：

```bash
docker compose -f docker/docker-compose.yml down -v
```

## 4. Docker 生产模式

```bash
docker compose \
  -f docker/docker-compose.yml \
  -f docker/docker-compose.prod.yml \
  up -d --build
```

`docker-compose.prod.yml` 覆盖关键差异：

| 项 | dev | prod |
|---|---|---|
| backend image | Dockerfile.dev | Dockerfile (无 build-essential 残留依赖最小化在镜像构建时) |
| backend command | `uvicorn ... --reload --reload-dir /app/app` | `uvicorn ...`（无 reload） |
| backend volumes | `../backend:/app`, `../data:/data` | 全部清空（`volumes: []`） |
| frontend image | Dockerfile.dev (vite) | Dockerfile (multi-stage → nginx) |
| frontend command | `vite --host 0.0.0.0` | `nginx -g daemon off;` |
| `DEMO_AUTH_OPTIONAL` | true | false |

生产前 `.env` 必改项见 `14-安全与鉴权.md §8`。

### 4.1 镜像构建优化

- 都启用 `.dockerignore`，避免 `node_modules` / `__pycache__` / `.git` 进入镜像；
- 后端 Dockerfile 把 `requirements.txt` 单独 COPY 一层 + `pip install`，依赖未变时层级缓存命中；
- 前端 Dockerfile 同理对 `package.json` 做依赖层。

### 4.2 nginx 反代

`frontend/nginx.conf` 把 `/api/` 反向代理到 `backend:8000/api/`，前端构建产物落在 `/usr/share/nginx/html`。如需 HTTPS、HSTS、CSP，在此文件追加 `server` 块或抽离独立 `nginx.prod.conf`。

## 5. 数据库迁移（Alembic）

### 5.1 目录结构

```
backend/
├── alembic.ini                 # 主配置（sqlalchemy.url 默认指向 PG，env.py 会覆盖为 settings）
└── alembic/
    ├── env.py                  # 通过 settings.DATABASE_URL_SYNC 注入
    └── versions/
        └── 0001_initial.py     # 基线：调用 Base.metadata.create_all
```

### 5.2 常用命令

```bash
cd backend
alembic upgrade head             # 应用所有迁移到当前 schema
alembic current                  # 查看当前版本
alembic history -v               # 查看迁移历史

# 增量开发：修改 ORM 后生成迁移
alembic revision --autogenerate -m "add process_params"
# review backend/alembic/versions/<rev>_add_process_params.py
alembic upgrade head             # 应用

# 回滚
alembic downgrade -1
alembic downgrade <rev>
```

### 5.3 SQLite 与 Alembic 的关系

- SQLite 模式下 `app/database.py: init_db()` 自动 `create_all`，alembic **不参与**（避免 demo 启动卡顿）；
- 切回 PG 时如果数据库已经被 `create_all` 建表，需要先 `alembic stamp head` 把版本号刻到 alembic_version 表里再继续 `revision/upgrade`，否则会重复建表报错。

## 6. 种子数据

来源：`data/seed/*.json`，共 16 份（factories / workshops / production_lines / equipment / sensors / products / materials / suppliers / customers / workers / sales_orders / work_orders / inspections / defects / spc_points / sensor_readings）。

### 6.1 自动注入（仅 SQLite）

`init_db()` 完成 `create_all` 后会查 `factories` 是否为空；若为空则调用 `_seed_sqlite` 批量插入，sensor_readings 按 5000 条/批。日志：

```
[INFO] app.database: Seeding SQLite from /app/../../data/seed ...
[INFO] app.database: Seeded 12345 rows
```

### 6.2 手动注入（PG / Neo4j）

```bash
cd <repo-root>
python scripts/seed_data.py        # 实现细节见脚本顶部注释
```

如需重建图谱：

```python
from app.services.graph_service import graph_service
import json
seed = {k: json.load(open(f"data/seed/{k}.json")) for k in ["factories","workshops",...]}
await graph_service.build_from_seed(seed)
```

> 注意：当前 `data/simulators/generate_sample_data.py` 与 `scripts/seed_data.py` 与 `database._seed_sqlite` 三套逻辑并存，建议未来合并为一个 `scripts/seed.py` 同时支持 SQLite/PG/Neo4j。

## 7. 健康检查与可观测性

| 端点 | 用途 |
|---|---|
| `GET /` | 元信息（appname / version） |
| `GET /health` | 简单存活探针，用于 docker healthcheck |
| `GET /docs` | Swagger UI |
| `GET /redoc` | ReDoc UI |
| `GET /openapi.json` | OpenAPI schema |

`backend` 容器自带：

```
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
  CMD curl -fsS http://localhost:8000/health || exit 1
```

集中日志：

- 当前所有日志统一走 stdout（`uvicorn` + `app.core.logging.setup_logging`），可直接被 docker / k8s 采集；
- 推荐 fluent-bit / promtail → ELK / Loki，按 `name` 过滤模块。

指标：

- 暂未集成 Prometheus；后续可接 `prometheus-fastapi-instrumentator`，挂在 `app/main.py` 即可暴露 `/metrics`。

## 8. 常见故障与回滚

### 8.1 后端启动失败

| 症状 | 排查 |
|---|---|
| `ModuleNotFoundError: pydantic_settings` | `pip install -r requirements.txt` 没跑或装到错误 venv |
| `[INFO] Database backend: sqlite` 但本地 PG 实际可用 | 设了 `POSTGRES_HOST` 但驱动 import 失败；查 `asyncpg` 是否安装 |
| 启动卡死在「Seeding...」 | 大量 `sensor_readings`，等几秒；或用 `LOG_LEVEL=DEBUG` 看进度 |

### 8.2 鉴权报错

| 症状 | 处理 |
|---|---|
| 前端反复跳转 `/login` | F12 → Network 看 `/auth/me` 状态；多半是 token 与 SECRET_KEY 不一致（生产换了 KEY 但前端缓存了旧 token），清 localStorage |
| 401 但日志无错误 | `DEMO_AUTH_OPTIONAL=true` 会静默把请求当 guest；调成 `false` 复现 |

### 8.3 Docker

| 症状 | 处理 |
|---|---|
| `manufoundry-frontend` 无 HMR | 确认用的是 `Dockerfile.dev`（默认 compose 已切换），且 `volumes` 挂的是 `../frontend:/app` |
| `Cannot connect to APOC plugin` | Neo4j 首次启动联网失败；在 compose 内注释 `NEO4J_PLUGINS` 或离线挂 plugin jar |
| `pg_isready` 一直未通过 | 检查端口冲突：`netstat -ano | findstr 5432` |

### 8.4 回滚

| 范围 | 命令 |
|---|---|
| Schema | `alembic downgrade -1`（或具体 revision） |
| 容器镜像 | 重新 `docker compose pull` 上一版本 tag 然后 `up -d` |
| Git | `git revert <commit>` 或 `git reset --hard <tag>` 后重新部署 |

灾难恢复：

- PG 用 `pg_dump` + 定时备份（推荐 every 6h + WAL 流复制）；
- Neo4j 用 `neo4j-admin database backup`；
- Redis 数据为缓存，可丢；
- 上传文件（如 reports/exports）目前保存在容器内 `/app`，建议挂 `s3:` 或外置 volume。
