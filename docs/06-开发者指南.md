本文档面向开发者，介绍如何搭建开发环境、理解项目架构，以及在现有代码基础上进行扩展开发。

## 1. 开发环境搭建

### 1.1 系统要求

| 工具 | 最低版本 | 推荐版本 | 用途 |
|------|---------|---------|------|
| Python | 3.11 | 3.11+ | 后端运行时 |
| Node.js | 18 | 20 LTS | 前端构建 |
| npm | 9 | 10+ | 前端包管理 |
| Git | 2.30 | 最新 | 版本控制 |

推荐 IDE：**VS Code** 或 **Cursor**，建议安装以下扩展：
- Python + Pylance（后端开发）
- ESLint + Prettier（前端代码规范）
- Thunder Client 或 REST Client（API 调试）

### 1.2 后端环境搭建

```bash
# 1. 进入后端目录
cd backend

# 2. 创建并激活虚拟环境
python -m venv venv
# Windows:
venv\Scripts\activate
# macOS/Linux:
source venv/bin/activate

# 3. 安装依赖
pip install -r requirements.txt

# 4. 安装 SQLite 异步驱动（用于本地开发）
pip install aiosqlite

# 5. 启动后端（无需 PostgreSQL/Neo4j，自动使用 SQLite + Mock 数据）
uvicorn app.main:app --reload --port 8000
```

启动成功后访问：
- API 文档（Swagger）：http://localhost:8000/docs
- 健康检查：http://localhost:8000/health

### 1.3 前端环境搭建

```bash
# 1. 进入前端目录
cd frontend

# 2. 安装依赖
npm install

# 3. 启动开发服务器
npm run dev
```

前端默认运行在 http://localhost:5173，API 请求会代理到后端 8000 端口。

### 1.4 Docker 全栈启动（可选）

如果需要完整环境（PostgreSQL + Neo4j + Redis + 后端 + 前端）：

```bash
# 在项目根目录下
cd docker
docker-compose up --build
```

启动后服务分布：
| 服务 | 地址 | 说明 |
|------|------|------|
| 前端 | http://localhost:3000 | Nginx 反向代理 |
| 后端 API | http://localhost:8000 | FastAPI |
| PostgreSQL | localhost:5432 | 关系数据库 |
| Neo4j Browser | http://localhost:7474 | 图数据库管理界面 |
| Redis | localhost:6379 | 缓存/消息队列 |

### 3.4 数据库层（`app/database.py`）

核心机制是 **自动检测**：

```python
# 检测 PostgreSQL 驱动是否可用
try:
    import asyncpg
    _engine = create_async_engine(settings.DATABASE_URL, ...)
    DB_TYPE = "postgresql"
except Exception:
    _engine = create_async_engine("sqlite+aiosqlite:///...", ...)
    DB_TYPE = "sqlite"
```

- 若 `asyncpg` 可用且 PostgreSQL 可连接 → 使用 PostgreSQL
- 否则自动降级到 SQLite，并在首次启动时建表 + 填充种子数据

Neo4j 同理：
```python
try:
    neo4j_driver = AsyncGraphDatabase.driver(settings.NEO4J_URI, ...)
except Exception:
    neo4j_driver = None  # 降级为 Mock
```

### 3.5 `_try_db()` 模式

这是整个项目最核心的设计模式。每个 API 模块都实现了这个模式：

```python
async def _try_db(fn):
    """尝试真实数据库查询，失败返回 None。"""
    try:
        from app.database import AsyncSessionLocal
        async with AsyncSessionLocal() as session:
            return await fn(session)
    except Exception:
        return None

@router.get("/overview")
async def get_overview():
    # 1. 尝试真实数据库查询
    result = await _try_db(lambda db: _query_overview(db))
    if result is not None:
        return result
    # 2. 数据库不可用时，返回预定义的 Mock 数据
    return MOCK_OVERVIEW
```

这确保了：
- **离线开发**：无需安装 PostgreSQL/Neo4j，SQLite 甚至不需要（直接 Mock）
- **演示场景**：任何环境都能展示完整功能
- **平滑迁移**：连接真实数据库后自动切换到真实数据

### 4.2 架构设计

```
App.tsx
├── Layout (Ant Design)
│   ├── Sider（侧边菜单，9 个入口）
│   ├── Header（标题栏）
│   └── Content
│       └── ErrorBoundary（崩溃隔离）
│           └── Suspense（懒加载）
│               └── Routes（9 个页面路由）
```

**关键设计决策**：

1. **Lazy Loading**：所有 9 个页面通过 `React.lazy()` 按需加载，减少首屏体积
2. **ErrorBoundary**：包裹每个路由页面，单个页面崩溃不影响整个应用
3. **Suspense + Spin**：懒加载时显示加载动画

```tsx
// 页面懒加载
const DashboardPage = lazy(() => import('./pages/Dashboard'));
const MaintenancePage = lazy(() => import('./pages/Maintenance'));
// ...

// 路由使用 ErrorBoundary + Suspense
<ErrorBoundary>
  <Suspense fallback={<PageLoader />}>
    <Routes>
      <Route path="/" element={<DashboardPage />} />
      <Route path="/maintenance" element={<MaintenancePage />} />
      // ...
    </Routes>
  </Suspense>
</ErrorBoundary>
```

## 7. Mock 数据模式说明

### 7.1 为什么需要 Mock 数据

| 场景 | 说明 |
|------|------|
| 离线开发 | 开发者可能没有安装 PostgreSQL/Neo4j |
| 快速演示 | 向客户展示功能时无需搭建完整环境 |
| CI/CD | 自动化测试不依赖外部数据库 |
| 降级容错 | 生产环境数据库临时故障时仍可提供基础响应 |

### 7.2 `_try_db()` 工作流程

```
API 请求到达
    │
    ▼
_try_db(查询函数)
    │
    ├─ 成功 → 返回真实数据 → 响应给客户端
    │
    └─ 失败（异常）→ 返回 None
                         │
                         ▼
                  返回 Mock 数据 → 响应给客户端
```

**关键代码**（每个 API 模块中重复出现）：

```python
async def _try_db(fn):
    try:
        from app.database import AsyncSessionLocal
        async with AsyncSessionLocal() as session:
            return await fn(session)
    except Exception:
        return None
```

注意：`except Exception` 捕获所有异常（包括 ImportError、连接错误、查询错误），确保永远不会因为数据库问题导致 API 崩溃。

### 7.3 图谱模块的 `_try_neo4j()`

图谱模块（`graph.py`）使用类似的 `_try_neo4j()` 函数，但操作的是 Neo4j 驱动而非 SQLAlchemy Session：

```python
def _try_neo4j(fn):
    try:
        from app.database import get_neo4j
        # 获取 Neo4j session 并执行查询
        ...
    except Exception:
        return None
```

### 7.4 如何为新端点添加 Mock 数据

1. 在 API 文件顶部定义 Mock 常量：

```python
MOCK_MY_DATA = [
    {"id": 1, "name": "示例数据", "value": 100},
    {"id": 2, "name": "示例数据2", "value": 200},
]
```

2. 在端点函数中遵循 `_try_db` → `if result is not None` → `return MOCK` 三段式：

```python
@router.get("/my-endpoint")
async def my_endpoint():
    async def _query(db):
        # 真实数据库查询逻辑
        ...

    result = await _try_db(_query)
    if result is not None:
        return result
    return {"data": MOCK_MY_DATA}
```

3. Mock 数据应尽量贴近真实数据的结构和数值范围，确保前端展示一致。

## 8. 代码规范

### 8.1 Python 后端规范

| 规范 | 说明 |
|------|------|
| 代码风格 | 遵循 PEP 8 |
| 类型注解 | 所有函数参数和返回值必须加 type hints |
| 异步优先 | 所有数据库 I/O 使用 `async/await`，避免阻塞 |
| 模型定义 | 使用 SQLAlchemy 2.0 `Mapped[]` 语法 |
| API 定义 | 使用 FastAPI 路由装饰器 + Pydantic 模型 |
| 导入顺序 | 标准库 → 第三方库 → 本地模块 |

**示例**：

```python
# 好的写法
async def get_equipment_by_id(equipment_id: int) -> dict | None:
    """根据 ID 获取设备信息."""
    ...

# 避免
def get_equipment(id):  # 缺少类型注解
    ...
```

### 8.2 TypeScript 前端规范

| 规范 | 说明 |
|------|------|
| 严格模式 | `tsconfig.json` 启用 `strict: true` |
| Interface 优先 | 数据结构优先用 `interface` 定义，而非 `type` |
| 函数组件 | 使用函数组件 + Hooks，不使用 Class 组件 |
| API 调用 | 统一通过 `services/api.ts`，不在组件中直接写 axios |
| 状态管理 | 简单状态用 `useState`，跨组件共享用 Zustand |

### 8.3 Git 提交规范

使用 Conventional Commits 格式：

```
<type>(<scope>): <subject>

<body>
```

类型（type）：

| 类型 | 说明 |
|------|------|
| feat | 新功能 |
| fix | Bug 修复 |
| docs | 文档更新 |
| style | 代码格式（不影响逻辑） |
| refactor | 重构 |
| test | 测试 |
| chore | 构建/工具变更 |

示例：
```
feat(maintenance): 添加设备健康评分趋势图
fix(quality): 修复 SPC 控制图 Cpk 计算精度问题
docs: 更新开发者指南
```
