# 业务交互表的表单逻辑

Last updated: 2026-06-05

本文解释“业务交互表”在当前平台中的定位、配置方式、运行时读写链路、权限边界和后续扩展方向。它面向产品、实施、测试和研发协作，不替代底层架构文档。

## 1. 定位

业务交互表不是一张固定数据库表，也不是简单的数据列表。它是由平台表单元数据驱动的运行时工作台：

```text
表单定义
  -> 字段元数据
  -> 列表 / 表单 / 筛选配置
  -> 动作与权限
  -> 工作流绑定
  -> 运行态业务记录
```

当前典型入口是：

```text
/dynamic/:slug
```

前端会把 `slug` 解析为平台表单 ID 或表单编码，读取已发布 schema，然后按字段、视图和权限配置渲染列表、筛选器、录入表单和操作按钮。业务数据不会默认为每个表单创建独立物理表，而是统一存入：

```text
dynamic_records.data
```

这种设计让业务可以快速配置新的交互表，而不必每次都修改数据库结构、开发专用 CRUD 页面、重新发布专用接口。

## 2. 配置态

业务交互表的配置分为四层。

第一层是表单身份：

- 表单名称和编码。
- 业务对象或模型绑定。
- 说明、状态、发布版本。
- 所属应用和菜单入口。

第二层是字段结构：

- 字段名和显示名。
- 存储字段类型。
- 业务类型和控件类型。
- 是否必填、是否列表展示、是否表单展示。
- 是否可搜索、可排序。
- 枚举值、默认值、校验配置和 UI 配置。
- 编码字段通过 `ui_config.businessType = code` 和 `encodingRule` 表达，存储类型仍按字符串处理。

第三层是页面交互：

- 表格列。
- 筛选器。
- 分页大小。
- 排序字段。
- 空值展示。
- 状态标签。
- 新增、编辑、删除、导入、导出等动作。

第四层是业务控制：

- 表单级权限。
- 字段级可见、可编辑、可导出权限。
- 工作流节点下的字段差异。
- 发布版本快照。
- 审计记录。

这四层共同决定一张业务交互表最终长什么样、谁能看、谁能改、数据能否保存，以及后续是否进入流程。

## 3. 运行态加载

进入 `/dynamic/:slug` 后，运行页会先加载表单定义。数字 `slug` 按表单 ID 查询，非数字 `slug` 尝试匹配表单编码。平台表单读取时优先使用已发布 schema，使运行页面对稳定版本，而不是正在编辑的草稿。

然后前端生成页面结构：

- `visible_in_list` 决定哪些字段进入表格列。
- `visible_in_form` 决定哪些字段进入新增/编辑表单。
- `searchable` 决定哪些字段参与搜索。
- `viewConfig.table.columns` 可以覆盖列宽、排序、标签和展示方式。
- `viewConfig.filters` 生成结构化筛选器。
- `runtime_permissions` 控制当前用户可见的动作按钮。
- `runtime_field_permissions` 控制当前用户在字段级别的可见、可编辑和可导出能力。

最后拉取业务记录：

```text
GET /api/v1/forms/{form_id}/records
```

常用参数包括：

- `page`
- `page_size`
- `search`
- `filters`
- `sort_field`
- `sort_order`

后端会先检查当前用户是否有 `view` 权限，再按字段可见权限裁剪返回数据。前端隐藏字段只是体验优化，真正的权限边界在后端。

## 4. 新增与编辑

新增记录时，运行页打开表单弹窗，按 `visible_in_form` 和字段权限渲染控件。提交前，前端先执行 Ant Design 表单校验，再把字段值转换成后端 payload。

新增调用：

```text
POST /api/v1/forms/{form_id}/records
```

编辑调用：

```text
PUT /api/v1/forms/{form_id}/records/{record_id}
```

后端保存前会执行三类检查：

- 表单是否属于当前租户。
- 用户是否拥有 `create` 或 `edit` 权限。
- 提交字段是否在当前用户可编辑字段集合内。

随后才会执行字段级数据校验，包括未定义字段拦截、必填校验、基础类型校验、枚举范围校验等。编辑时不会简单覆盖整条 JSON，而是把提交字段合并到原记录里，再用合并后的结果校验，避免局部编辑误删未展示字段。

## 5. 删除

删除记录调用：

```text
DELETE /api/v1/forms/{form_id}/records/{record_id}
```

后端会检查 `delete` 权限，并把记录标记为删除时间，而不是直接物理删除。默认列表不返回已删除记录。这个逻辑适合业务系统，因为交互表中的记录通常承载流程、审批、证据和审计责任。

## 6. 权限模型

业务交互表的权限分两层。

表单级权限决定用户能不能执行动作：

- `view`
- `create`
- `edit`
- `delete`
- `import`
- `export`
- `configure`
- `approve`

字段级权限决定用户对具体字段能不能：

- 可见。
- 可编辑。
- 可导出。
- 在特定流程节点下必填。

运行页会读取 `runtime_permissions` 控制按钮是否出现，也会读取 `runtime_field_permissions` 控制字段是否展示或可编辑。但后端接口每次仍会独立检查权限，防止用户绕过页面直接调用 API。

## 7. 发布版本

表单配置不是保存后立刻改变业务运行。当前平台支持发布版本快照：发布时把表单、字段、布局、动作、权限和工作流绑定整理成 snapshot。

运行页读取已发布 schema 的意义是：

- 正在编辑的草稿不会影响业务录入。
- 已有业务记录能保留当时的 `schema_version`。
- 表单变更可以被审查、发布和追溯。
- 后续可以扩展版本回看、版本差异和迁移策略。

因此，业务交互表的正确心智不是“改字段立刻改生产表”，而是“配置草稿经过发布后成为新的运行态版本”。

## 8. 示例

以“设备告警处理表”为例，字段可以包括：

| 字段 | 类型 | 列表展示 | 表单录入 | 业务含义 |
| --- | --- | --- | --- | --- |
| 告警编号 | 编码 | 是 | 是 | 唯一识别一条告警 |
| 标题 | 文本 | 是 | 是 | 快速说明问题 |
| 设备 | 关联选择 | 是 | 是 | 绑定设备对象 |
| 等级 | 枚举 | 是 | 是 | 普通、重要、严重 |
| 责任人 | 人员选择 | 是 | 是 | 指派处理人 |
| 截止时间 | 日期 | 是 | 是 | 处理时限 |
| 状态 | 枚举 | 是 | 是 | 待处理、处理中、已关闭 |
| 处理结论 | 多行文本 | 否 | 是 | 关闭时填写 |
| 附件证据 | 附件 | 否 | 是 | 现场图片或日志 |

列表页先帮助用户筛选“严重 + 逾期 + 我的责任”记录。进入编辑后，责任人补充处理结论和附件。保存时，后端校验字段类型和权限。提交时，后续可以绑定流程，让主管审批关闭。

## 9. 当前边界

当前实现已经具备平台表单运行态的核心闭环，但仍有一些边界需要明确：

- 创建字段不会自动创建物理数据库列。
- 字段联动、复杂条件显示和规则编排已有设计入口，但并非所有运行态行为都已完整落地。
- 导入、导出、审批等动作已有模型和权限基础，具体业务动作需要按场景继续接入。
- 高容量动态记录仍需要更完整的索引、查询优化和归档策略。
- 字段级权限已进入运行态摘要和写入检查，复杂流程节点下的字段差异仍需要继续产品化。

## 10. 对外表述

可以用一句话解释：

```text
业务交互表是由平台表单元数据驱动的运行态工作台，它把列表、筛选、录入、权限、流程和审计组合成一个可配置的业务闭环。
```

对研发说明时，可以换成：

```text
业务交互表不是一张新物理表，而是 forms / form_fields / form_layouts / form_actions / form_permissions / workflow_bindings 与 dynamic_records 的运行态组合。
```
