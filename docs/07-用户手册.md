> 版本：0.1.0 | 最后更新：2026-04-21

## 系统概述

ManuFoundry 是一个面向制造业的数据操作系统，提供从数据采集、本体建模、知识图谱、预测性维护到质量管理的全链路功能。

**访问地址**：启动后访问 http://localhost:3000（Docker 部署）或 http://localhost:5173（开发模式）

**界面布局**：


## 1. 运营总览（Dashboard）

运营总览是系统的首页，提供工厂运营状况的全局视图。

### 1.1 KPI 卡片

页面顶部展示 6 个关键绩效指标卡片：

| KPI | 说明 | 计算方式 |
|-----|------|---------|
| 设备总数 | 工厂内全部设备数量 | 直接计数 |
| 运行中 | 当前处于运行状态的设备数 | status = "running" 的设备数 |
| 平均健康评分 | 所有设备的健康评分均值 | 各设备 health_score 的算术平均值（满分 100） |
| 工单总数 | 当前所有工单数量 | 全部 WorkOrder 计数 |
| 缺陷数 | 质量缺陷总数 | Defect 表总记录数 |
| 利用率 | 设备利用率百分比 | 运行中设备数 / 总设备数 |

每个卡片使用 Ant Design Statistic 组件展示数值，配有对应图标和颜色标识。

### 1.2 OEE 堆叠柱状图

展示各产线的 **设备综合效率（OEE）** 分解：

- **可用率（Availability）**：设备实际运行时间占计划时间的比例
- **性能率（Performance）**：实际产出速度占理论速度的比例
- **质量率（Quality）**：合格品占全部产出的比例

OEE = 可用率 x 性能率 x 质量率

图表采用堆叠柱状图形式，每根柱子代表一条产线，三种颜色分别对应三个分项。柱子顶部的黑色标记线表示该产线的 OEE 目标值。

**操作**：
- 鼠标悬停柱子可查看具体数值
- 图例可点击切换显示/隐藏

### 1.3 近7日生产趋势

展示最近 7 天（可调）的生产数据折线图：

- **计划产量**（蓝色虚线）：每日计划生产数量
- **实际产量**（绿色实线）：每日实际完成数量
- **合格率**（橙色实线）：当日合格品比率（右侧 Y 轴）

**操作**：
- 可通过 API 参数调整天数范围（1~90 天）

### 1.4 告警列表

显示当前需要关注的告警信息，按严重程度排列：

| 字段 | 说明 |
|------|------|
| 严重程度 | critical（红色）= 健康评分 < 40，warning（黄色）= 健康评分 40~70 |
| 类型 | 当前为 equipment_health（设备健康预警） |
| 标题 | 如「设备 CNC-003 健康评分偏低」 |
| 详情 | 如「当前健康评分: 65.3」 |
| 时间 | 告警产生时间 |

**操作**：
- 默认显示最近 20 条，可通过参数调整（最多 100 条）

## 2. 数据源管理（DataSource）

管理所有外部数据源的连接配置和同步状态。

### 2.1 数据源列表

系统预置 8 个数据源：

| 名称 | 类型 | 状态 | 说明 |
|------|------|------|------|
| MES生产执行系统 | mes | active | 制造执行系统数据 |
| SAP ERP系统 | erp | active | 企业资源计划数据 |
| IoT传感器网关 | iot | active | 传感器实时数据 |
| PLC控制器-产线A | plc | active | 可编程逻辑控制器 |
| 质量检测API | api | paused | 第三方质量检测接口 |
| WMS仓储数据库 | database | active | 仓储管理系统 |
| SCADA数据采集 | plc | error | 数据采集与监控 |
| 能源监控平台 | iot | active | 能源消耗监测 |

**状态含义**：
- `active`：正常运行
- `paused`：已暂停同步
- `error`：连接异常
- `syncing`：正在同步中

**操作**：
- **筛选**：可按类型（mes/erp/iot/plc/api/database）或状态过滤
- **新增**：点击「新增」按钮，填写名称、类型、连接配置后创建

### 2.2 测试连接

点击某个数据源的「测试连接」按钮，系统将尝试连接并返回结果：

- **连接状态**：success / failed
- **延迟**：连接响应时间（ms）
- **记录数**：可读取的数据条数

### 2.3 触发同步

点击「同步」按钮可手动触发数据同步。同步后：
- 数据源状态变为 `syncing`
- 同步完成后恢复为 `active`
- 更新 `last_sync` 时间戳

### 2.4 数据预览

点击「预览」查看数据源的采样数据，默认显示 10 条记录。展示字段包括：
- id、timestamp、value、status 等通用字段
- 总记录数

## 3. 本体建模器（Ontology）

定义和管理制造业知识图谱的实体类型和关系类型，是系统数据模型的核心。

### 3.1 实体类型列表

系统定义了 13 种实体类型：

| 实体类型 | 中文名 | 核心属性 | 出向关系 |
|---------|--------|---------|---------|
| Factory | 工厂 | name, location, capacity, status | CONTAINS → Workshop |
| Workshop | 车间 | name, area, workshop_type | CONTAINS → ProductionLine |
| ProductionLine | 产线 | name, capacity, oee_target, status | CONTAINS → Equipment, PRODUCES → Product |
| Equipment | 设备 | name, model, manufacturer, status, health_score | FEEDS → Sensor |
| Sensor | 传感器 | name, sensor_type, unit, sampling_rate | 无 |
| Product | 产品 | name, sku, category, specs | REQUIRES → Material |
| Material | 物料 | name, material_type, specs, safety_stock | 无 |
| Supplier | 供应商 | name, location, rating, lead_time_days | SUPPLIES → Material |
| Customer | 客户 | name, industry, region | 无 |
| Worker | 工人 | name, role, department | ASSIGNED_TO → Equipment/WorkOrder |
| SalesOrder | 销售订单 | order_no, customer, product, quantity, due_date | — |
| WorkOrder | 工单 | order_no, line, quantity, status | — |
| Warehouse | 仓库 | name, location, capacity, utilization | — |

**操作**：
- 点击实体类型卡片可查看详细定义（属性列表、允许关系等）

### 3.2 实体实例浏览

选择某种实体类型后，可浏览该类型下的所有实例数据。

- 支持分页（每页 20 条）
- 显示该实体类型对应数据库表的全部字段
- 日期字段自动格式化

例如选择「Equipment」可看到所有设备的名称、型号、制造商、状态、健康评分等信息。

### 3.3 关系类型查看

系统定义了 10 种关系类型：

| 关系 | 说明 | 典型用法 |
|------|------|---------|
| CONTAINS | 包含 | Factory → Workshop → ProductionLine → Equipment |
| PRODUCES | 生产 | ProductionLine → Product |
| REQUIRES | 需要 | Product → Material |
| SUPPLIES | 供应 | Supplier → Material |
| INSPECTS | 检验 | Inspection → Material |
| MAINTAINS | 维护 | Worker → Equipment |
| FEEDS | 供数 | Equipment → Sensor |
| ASSIGNED_TO | 分配 | Worker → Equipment/WorkOrder |
| STORED_IN | 存储 | Material → Warehouse |
| SHIPS_TO | 发货 | Shipment → Warehouse |

## 4. 关系图谱（GraphExplorer）

以可视化图谱形式展示实体之间的关联关系。

### 4.1 图谱统计

页面顶部展示图谱全局统计：

- **总节点数**：当前图谱中的实体总数（约 65 个）
- **总关系数**：实体间的关系总数（约 128 条）
- **节点分布**：各类型节点的数量（Equipment 最多，约 20 个）
- **关系统计**：各类型关系的数量（CONTAINS 最多，约 45 条）

### 4.2 邻居节点探索

选择一个实体节点后，可查看其直接关联的邻居节点：

**操作步骤**：
1. 输入实体 ID
2. 系统返回所有相邻节点，包含：
   - 邻居节点信息（标签、名称、属性）
   - 关系类型（CONTAINS, PRODUCES 等）
   - 关系方向（outgoing = 出向, incoming = 入向）

**示例**：查看 Factory(id=1) 的邻居，可发现其包含的 Workshop、产线等信息。

### 4.3 最短路径查询

查找两个实体之间的最短关联路径。

**操作步骤**：
1. 输入起始实体 ID（如 Supplier=8）
2. 输入目标实体 ID（如 Equipment=4）
3. 设置最大跳数（1~10，默认 6）
4. 系统返回路径上的所有节点和关系

**用途**：分析供应链风险传导路径、追溯质量问题传播链路等。

### 4.4 子图提取

以某个实体为中心，提取指定深度的子图。

**操作步骤**：
1. 输入中心实体 ID
2. 设置深度（1~5，默认 2）
3. 设置最大节点数（10~500，默认 100）
4. 系统返回子图中的所有节点和关系

**用途**：聚焦查看某个设备、产品或供应商的关联网络。

### 4.5 Cypher 查询

支持直接执行 Neo4j Cypher 查询语言（连接 Neo4j 时可用）。

**示例查询**：
```cypher
// 查找所有设备及其传感器
MATCH (e:Equipment)-[:FEEDS]->(s:Sensor) RETURN e, s

// 查找某产品的完整供应链
MATCH (p:Product)-[:REQUIRES]->(m:Material)<-[:SUPPLIES]-(sup:Supplier)
RETURN p.name, m.name, sup.name
```

## 5. 数据管线（Pipeline）

管理数据从采集到入库的自动化流程。

### 5.1 管线列表

系统预置 6 条数据管线：

| 名称 | 说明 | 状态 | 调度 |
|------|------|------|------|
| MES数据同步管线 | 从MES系统同步生产数据到数据湖 | active | 每5分钟 |
| IoT传感器ETL | 实时传感器数据清洗与聚合 | active | 每1分钟 |
| 质量检测数据管线 | 三坐标测量数据自动采集与分析 | active | 每小时 |
| ERP物料同步 | SAP物料主数据同步 | paused | 每6小时 |
| 设备预测维护模型训练 | 基于历史数据训练故障预测模型 | draft | 手动 |
| 供应链库存快照 | 每日库存快照生成与报表推送 | active | 每天凌晨2点 |

**状态含义**：
- `active`：按调度自动运行
- `paused`：暂停运行
- `draft`：草稿，未激活
- `running`：正在执行

### 5.2 创建新管线

点击「新建管线」按钮，填写：

| 字段 | 说明 | 示例 |
|------|------|------|
| 名称 | 管线名称 | 能耗数据同步 |
| 描述 | 功能描述 | 从能源监控平台同步能耗数据 |
| 配置 | JSON 格式的步骤定义 | `{"steps": [{"type": "extract", "source": "energy"}, {"type": "transform"}, {"type": "load"}]}` |
| 调度 | Cron 表达式 | `*/10 * * * *`（每10分钟） |

新建管线默认状态为 `draft`。

### 5.3 手动执行管线

点击某条管线的「执行」按钮：

- 管线状态变为 `running`
- 系统模拟执行过程
- 执行完成后显示：
  - 处理的记录数（records_processed）
  - 开始和结束时间
  - 执行状态（completed / failed）

### 5.4 查看执行历史

点击管线详情可查看历史执行记录：

| 字段 | 说明 |
|------|------|
| Run ID | 执行记录 ID |
| 状态 | completed / failed / running |
| 处理记录数 | 本次处理的记录数量 |
| 开始时间 | 执行开始时刻 |
| 结束时间 | 执行完成时刻 |
| 错误信息 | 失败时的错误描述（如「连接MES系统超时」） |

## 7. 质量管理（Quality）

提供全面的质量数据分析、缺陷管理和追溯功能。

### 7.1 SPC 控制图

**统计过程控制（SPC）** 实时监控生产参数。

**支持的参数**：
- `temperature`（温度）
- `vibration`（振动）
- `pressure`（压力）
- `dimension`（尺寸）

**图表内容**：
- 折线图展示参数值随时间变化
- 上控制限（UCL）和下控制限（LCL）以红色虚线标识
- 中心线（CL）以绿色实线标识
- 超出控制限的点标记为红色（异常点）

**Cpk 值**：
每张 SPC 图自动计算 Cpk（过程能力指数）：
- Cpk >= 1.33：过程能力充足
- 1.0 <= Cpk < 1.33：过程能力勉强
- Cpk < 1.0：过程能力不足，需改善

**操作**：
- 选择参数类型
- 可按设备过滤
- 可调整时间范围（1~168 小时）

### 7.2 缺陷列表

记录所有检测发现的缺陷：

| 字段 | 说明 | 示例 |
|------|------|------|
| 缺陷类型 | 缺陷分类 | 尺寸超差、表面缺陷、裂纹、变形、硬度不足、毛刺、气孔 |
| 严重度 | critical / major / minor | critical |
| 描述 | 具体缺陷描述 | 外径Φ50+0.02实测Φ50.035 |
| 根因 | 导致缺陷的原因 | 刀具磨损导致进给偏差 |
| 纠正措施 | 已采取的措施 | 更换刀具并调整补偿值 |

**操作**：
- 可按严重度筛选（critical / major / minor）
- 支持分页浏览

### 7.3 帕累托分析

基于 **80/20 法则** 分析缺陷分布，帮助确定优先改善项：

| 缺陷类型 | 数量 | 占比 | 累计百分比 |
|---------|------|------|-----------|
| 尺寸超差 | 12 | 30.0% | 30.0% |
| 表面缺陷 | 8 | 20.0% | 50.0% |
| 裂纹 | 6 | 15.0% | 65.0% |
| 变形 | 5 | 12.5% | 77.5% |
| 硬度不足 | 4 | 10.0% | 87.5% |
| 毛刺 | 3 | 7.5% | 95.0% |
| 气孔 | 2 | 5.0% | 100.0% |

图表包含柱状图（各类型数量）和折线图（累计百分比），80% 累计线用红色虚线标识。

**分析结论**：尺寸超差 + 表面缺陷 + 裂纹三类占全部缺陷的 65%，应优先改善。

### 7.4 质量追溯链

追踪产品的完整生产过程质量记录。

**以产品追溯为例**，链条包含以下环节：

```
来料检验 → 工序1-切割 → 过程检验 → 工序2-焊接 → 工序3-组装 → 成品检验 → 包装出库
```

每个环节记录：
| 字段 | 说明 |
|------|------|
| 步骤名称 | 如「来料检验」「工序1-切割」 |
| 类型 | inspection / operation / shipment |
| 结果 | pass / fail / completed |
| 时间 | 执行时间 |

## 常见问题

### Q1: 启动后页面空白？
确认后端服务已启动（http://localhost:8000/health 返回 healthy），前端 API 代理配置正确。

### Q2: 数据全是 Mock 的？
系统在没有连接 PostgreSQL/Neo4j 时自动使用 Mock 数据。如需真实数据，请通过 Docker Compose 启动完整环境。

### Q3: 如何重置数据？
删除 `backend/manufoundry.db` 文件，重启后端服务，系统会自动重建 SQLite 数据库并填充种子数据。

### Q4: AI 助手回答不准确？
当前 AI 助手基于关键词匹配和预设模板，并非真正的大语言模型。如需接入真正的 AI，需在 `.env` 中配置 `OPENAI_API_KEY`。

### Q5: 图谱页面无法显示关系图？
图谱可视化需要 Neo4j 服务。未连接 Neo4j 时，图谱统计和查询仍可用（返回 Mock 数据），但 Cytoscape 可视化可能不显示。
