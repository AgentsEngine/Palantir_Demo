## 1. 概述

ManuFoundry 采用 **双数据库架构**，分别承载结构化业务数据和知识图谱关系数据：

| 维度 | 关系数据库 (SQL) | 图数据库 (Neo4j) |
|------|-------------------|-------------------|
| **引擎** | PostgreSQL（生产）/ SQLite（开发） | Neo4j（可选） |
| **ORM** | SQLAlchemy 2.0 (async) | py2neo /原生 Cypher |
| **模型数** | 26 个 ORM 模型 | 13 种节点标签 + 10 种关系类型 |
| **用途** | 业务 CRUD、事务处理 | 关系发现、路径查询、图谱可视化 |
| **同步** | 关系数据为主 | 关系数据同步到图节点（pg_id 映射） |

### 1.1 自动建表

首次启动时，`database.py` 中的 `init_db()` 函数会自动执行：

1. **建表** — 调用 `Base.metadata.create_all()` 创建全部 26 张表
2. **种子数据导入** — 检测 `Factory` 表是否为空，若为空则从 `data/seed/*.json` 导入全部种子数据
3. **批量导入** — 传感器读数（9360 条）以 5000 条为一批分批写入，避免内存溢出

```python
# app/database.py — 自动初始化流程
async def init_db():
    if DB_TYPE != "sqlite":
        return
    async with _engine.begin() as conn:
        await conn.run_sync(Base.metadata.create_all)

    async with AsyncSessionLocal() as session:
        count = await session.scalar(select(func.count(Factory.id)))
        if count == 0:
            await _seed_sqlite(session)
```

## 2. 关系数据模型

### 2.1 工厂族（层级包含关系）

工厂族描述从工厂到设备的四级层级结构，通过外键形成严格的 `Factory → Workshop → ProductionLine → Equipment` 包含链。

#### Factory（工厂）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | Integer | PK, 自增 | 工厂编号 |
| `name` | String(200) | NOT NULL | 工厂名称 |
| `location` | String(500) | NOT NULL | 地理位置 |
| `capacity` | Float | DEFAULT 0 | 产能上限 |
| `status` | String(50) | DEFAULT 'active' | 状态 |
| `description` | Text | NULLABLE | 描述 |
| `created_at` | DateTime | 自动生成 | 创建时间 |
| `updated_at` | DateTime | 自动更新 | 更新时间 |

**表名**: `factories` | **关联**: `workshops` (一对多)

#### Workshop（车间）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | Integer | PK, 自增 | 车间编号 |
| `name` | String(200) | NOT NULL | 车间名称 |
| `factory_id` | Integer | FK → factories.id | 所属工厂 |
| `area` | Float | DEFAULT 0 | 面积（平方米） |
| `workshop_type` | String(100) | DEFAULT 'production' | 车间类型（machining/welding/assembly/inspection/casting/surface/smt/testing） |

**表名**: `workshops` | **关联**: `factory` (多对一), `production_lines` (一对多)

#### ProductionLine（产线）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | Integer | PK, 自增 | 产线编号 |
| `name` | String(200) | NOT NULL | 产线名称 |
| `workshop_id` | Integer | FK → workshops.id | 所属车间 |
| `capacity` | Float | DEFAULT 0 | 产能（件/日） |
| `oee_target` | Float | DEFAULT 0.85 | OEE 目标值 |
| `status` | String(50) | DEFAULT 'running' | 状态（running/idle/maintenance） |

**表名**: `production_lines` | **关联**: `workshop` (多对一), `equipment_list` (一对多)

#### Equipment（设备）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | Integer | PK, 自增 | 设备编号 |
| `name` | String(200) | NOT NULL | 设备名称 |
| `line_id` | Integer | FK → production_lines.id | 所属产线 |
| `model` | String(200) | NOT NULL | 设备型号 |
| `manufacturer` | String(200) | NOT NULL | 制造商 |
| `install_date` | DateTime | NULLABLE | 安装日期 |
| `status` | String(50) | DEFAULT 'running' | 状态枚举：running / idle / maintenance / fault / offline |
| `health_score` | Float | DEFAULT 100.0 | 健康评分（0-100） |

**表名**: `equipment` | **关联**: `production_line` (多对一), `sensors` (一对多)

**EquipmentStatus 枚举**:

| 值 | 说明 |
|----|------|
| `running` | 运行中 |
| `idle` | 空闲 |
| `maintenance` | 维护中 |
| `fault` | 故障 |
| `offline` | 离线 |

## 3. Neo4j 图模型

### 3.1 节点标签

图数据库中的节点通过 `pg_id` 属性与关系数据库中的记录保持对应。

| 节点标签 | 中文名 | 关键属性 | 出发关系 |
|----------|--------|----------|----------|
| `Factory` | 工厂 | name, location, capacity, status | CONTAINS |
| `Workshop` | 车间 | name, area, workshop_type | CONTAINS |
| `ProductionLine` | 产线 | name, capacity, oee_target, status | CONTAINS, PRODUCES |
| `Equipment` | 设备 | name, model, manufacturer, status, health_score | FEEDS |
| `Sensor` | 传感器 | name, sensor_type, unit, sampling_rate | — |
| `Product` | 产品 | name, sku, category, specs | REQUIRES |
| `Material` | 物料 | name, material_type, specs, safety_stock | — |
| `Supplier` | 供应商 | name, location, rating, lead_time_days | SUPPLIES |
| `Customer` | 客户 | name, industry, region | — |
| `SalesOrder` | 销售订单 | — | — |
| `WorkOrder` | 工单 | — | — |
| `Inspection` | 检验 | — | — |
| `Defect` | 缺陷 | — | — |
| `Warehouse` | 仓库 | — | STORED_IN, SHIPS_TO |
| `Worker` | 工人 | name, role, department | ASSIGNED_TO |

### 3.2 关系类型

| 关系类型 | 中文名 | 起点 → 终点 | 属性示例 | 语义 |
|----------|--------|-------------|----------|------|
| `CONTAINS` | 包含 | Factory → Workshop → ProductionLine → Equipment | `{}` | 组织层级关系 |
| `PRODUCES` | 生产 | ProductionLine → Product | `{}` | 产线生产的产品 |
| `REQUIRES` | 需要 | Product → Material | `{quantity: 2.5}` | BOM 物料需求 |
| `SUPPLIES` | 供应 | Supplier → Material | `{}` | 供应商提供的物料 |
| `INSPECTS` | 检验 | Inspection → Material/Equipment | `{}` | 检验关联 |
| `MAINTAINS` | 维修 | Worker → Equipment | `{}` | 维护关系 |
| `FEEDS` | 采集 | Equipment → Sensor | `{}` | 设备挂载的传感器 |
| `ASSIGNED_TO` | 分配 | Worker → Equipment/WorkOrder | `{}` | 人员分配 |
| `STORED_IN` | 存储 | Material → Warehouse | `{}` | 库存位置 |
| `SHIPS_TO` | 发运 | Warehouse → Warehouse | `{}` | 物流流向 |

### 3.3 本体约束

`graph_models.py` 中定义了 `ENTITY_SCHEMAS` 字典，对每种实体类型做了严格约束：

```python
ENTITY_SCHEMAS = {
    "Factory": {
        "label": "工厂",
        "icon": "factory",
        "properties": {
            "name": {"type": "string", "required": True, "label": "工厂名称"},
            "location": {"type": "string", "required": True, "label": "地理位置"},
            "capacity": {"type": "float", "required": False, "label": "产能"},
            "status": {"type": "string", "required": False, "label": "状态"},
        },
        "outgoing_relations": ["CONTAINS"],
        "allowed_targets": {"CONTAINS": ["Workshop"]},
    },
    # ... 其余实体类似
}
```

每个实体定义了：
- **properties** — 允许的属性、类型、是否必填、中文标签
- **outgoing_relations** — 允许的出发关系类型
- **allowed_targets** — 每种关系允许的目标实体类型

### 3.4 Cypher 查询模板

系统内置 6 个常用 Cypher 查询模板（定义于 `CYPHER_TEMPLATES`）：

| 模板名 | 用途 |
|--------|------|
| `create_entity` | 创建节点 |
| `create_relation` | 创建关系 |
| `get_neighbors` | 获取邻居节点 |
| `shortest_path` | 最短路径查询 |
| `subgraph` | 子图提取 |
| `stats` | 图谱统计 |
